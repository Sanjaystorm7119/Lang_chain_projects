Building with the Claude API - Notes


Overview of Claude Models
Anthropic provides the Claude family of large language models with different variants optimized for different use cases:
• Claude models vary in size, cost, and capabilities (Haiku, Sonnet, Opus)
• General-purpose chat models suitable for most tasks
• Smaller, faster, cheaper models for cost-sensitive applications
• Models specialized for specific tasks like code generation
• Longer context window variants for processing large amounts of text
• Each model version offers different performance/cost tradeoffs


Accessing the API
You interact with Claude through HTTPS requests to Anthropic's API endpoints:
• Requests come from backend services, scripts, or tools (not directly from frontend)
• Each request includes your API key, model name, and configuration parameters
• Typical parameters include temperature, max_tokens, system prompts, and message history
• Responses return the model's output plus metadata (usage stats, response ID, timestamps)
• Use response metadata for logging, cost tracking, and observability


Getting an API Key
• API keys are created and managed in the Anthropic Console
• Each key is tied to your account and has specific usage limits
• Keys must be kept SECRET - never hardcode in frontend code or share publicly
• Always send keys in the Authorization header of HTTP requests
• Rotate keys periodically for security
• Revoke keys immediately if compromised


Making a Request
A typical Claude API request includes:
• Model: The specific Claude model you want to use (e.g., claude-3-sonnet-20240229)
• Messages: Array of message objects with roles (user, assistant, system)
• System prompt: Optional instructions defining behavior and constraints
• Temperature: Controls randomness (0 = deterministic, 1 = creative)
• Max tokens: Limits the length of the response
• Top_p: Controls diversity (nucleus sampling)
• Stop sequences: Text that stops generation


Response contains:
• The assistant's content/reply
• Stop reason (end_turn, max_tokens, etc.)
• Usage: Input tokens, output tokens, cache metrics
• ID and timestamp for tracking


Multi-Turn Conversations
Multi-turn chat is implemented by maintaining conversation history:
• Include full or truncated prior conversation on each new request
• Store conversation history server-side (database, cache, session storage)
• Format history as array of messages with prior user/assistant turns
• Model uses context to maintain coherence across multiple turns
• Consider token limits and truncate old messages if conversation is very long
• System prompt stays consistent across all turns


System Prompts
System prompts define the high-level behavior and constraints of Claude:
• System prompt is sent as a special message at the start of conversation
• Provides context like persona, tone, rules, and capabilities
• Example: "You are a Python expert. Always provide secure, production-ready code."
• Should be stable and explicit - not frequently changed
• Can include safety guidelines, domain expertise, output format requirements
• System prompts are the primary lever for controlling model behavior
• Different system prompts can dramatically change the same input's output


Temperature
Temperature controls the randomness and creativity of the model's responses:
• Low temperature (0.0-0.3): More deterministic, focused, reliable outputs
• Medium temperature (0.5-0.7): Balanced between consistency and variety
• High temperature (0.8-1.0): More diverse, creative, unpredictable outputs
• Temperature = 0 is the most deterministic (always produces same output for same input)
• For production systems, use low-to-moderate temperature for reliability
• For creative tasks (brainstorming, writing), higher temperature works better
• Default temperature is usually 1.0, but 0.7 is often a good starting point


Response Streaming
Streaming allows the client to receive tokens as they are generated:
• Instead of waiting for complete response, stream tokens in real-time
• Enables faster-feeling interfaces and better user experience
• Open streaming HTTP connection to the API
• Receive and render tokens/partial text as they arrive
• Close stream at completion
• Better for long responses where latency is noticeable
• Most client libraries (Python, JavaScript, etc.) support streaming
• Helps with perceived performance and responsiveness


Controlling Model Output
Various techniques to control the format and structure of Claude's responses:
• Clear instructions in prompt: "Respond in JSON format"
• Schema-based outputs: Define expected structure explicitly
• Max_tokens constraint: Limit response length
• Stop sequences: Tell model when to stop generating
• Role separation: Use system prompt, user input, assistant response clearly
• Combining multiple techniques is most reliable
• Always validate output matches expected format before using
• Combine system prompts with explicit formatting instructions for best results


Structured Data
Structured data features allow Claude to produce machine-readable outputs:
• Ask model to return JSON conforming to a specific schema
• Specify required fields, data types, and validation rules
• Useful for downstream automation and processing
• Example: Extract product title, price, and rating from text
• Example: Generate SQL query from natural language
• More reliable than parsing free-form text
• Can combine with JSON schema validation
• Saves engineering effort on parsing and error handling


Prompt Evaluation
• Assessing quality and effectiveness of prompts
• Methods for comparing different prompt versions
• Metrics for success (accuracy, consistency, relevance)
• A/B testing prompts to find the best approach
• Iterative refinement based on results


A Typical Eval Workflow
• Define clear success criteria and evaluation metrics
• Create a test dataset of representative examples
• Run Claude on each example with different prompts
• Compare outputs against ground truth or rubric
• Calculate success rate and other metrics
• Identify which prompts work best for different scenarios
• Document lessons learned


Generating Test Datasets
• Create diverse, representative test cases
• Cover normal cases, edge cases, and failure modes
• Label with expected correct outputs (ground truth)
• Ensure dataset quality and accuracy
• Test both common scenarios and unusual inputs
• Use real data when possible
• Build gradually as you encounter new cases


Running the Eval
• Execute all test cases with each prompt version
• Log inputs, outputs, and metadata for analysis
• Compare outputs to expected results
• Calculate success metrics (accuracy, F1 score, etc.)
• Handle errors and edge cases appropriately
• Track performance differences between versions


Model Based Grading
• Use Claude itself to grade responses (meta-evaluation)
• Create a grading rubric for Claude to follow
• Claude evaluates other Claude outputs
• Can scale evaluation to large datasets
• Still requires human review for validation
• Effective for nuanced quality assessment



Code Based Grading
• Validates AI-generated code responses for format, syntax, and task accuracy
• Format validation: Response should return only requested code type (Python, JSON, or Regex) without explanations
• Valid Syntax: Generated code should parse correctly as the intended language
• Task Following: Response should directly address what was asked and be accurate
• First two criteria handled by code grader, task following evaluated by model grader
• Syntax validation using try-except blocks:
 		 - validate_json(): Uses json.loads() to parse JSON, returns 10 if valid, 0 if JSONDecodeError
  	- validate_python(): Uses ast.parse() to check Python syntax, returns 10 if valid, 0 if SyntaxError
  	- validate_regex(): Uses re.compile() to validate regex, returns 10 if valid, 0 if re.error
• Dataset format requirements: Include "format" field to specify expected output type (python, json, or regex)
• Example dataset entry: {"task": "...", "format": "python"}

• Improving prompt clarity: Specify "Respond only with Python, JSON, or a plain Regex" and "Do not add any comments or commentary or explanation"
• Use pre-filled assistant message: add_assistant_message(messages, "```code") to encourage raw code output
• Combining scores: Average model grader score with syntax score: (model_score + syntax_score) / 2
• Equal weighting can be adjusted based on use case priorities
• Implementation: Run evaluation to get baseline score, which provides quantitative metrics for measuring prompt engineering progress


Prompt Engineering
• Process of improving prompts to get more reliable, higher-quality outputs
• Iterative improvement cycle: Set goal -> Write initial prompt -> Evaluate -> Apply techniques -> Re-evaluate
• PromptEvaluator class handles dataset generation and model grading
• Control concurrency with max_concurrent_tasks parameter (start with 3-5 to avoid rate limits)
• Keep number of test cases low (2-3) during development to speed up iteration
• Basic prompt establishes baseline, but likely produces poor results
• Evaluation provides numerical score and detailed HTML report
• Low initial scores typical (e.g., 2.3 out of 10) - goal is consistent improvement
• Detailed report shows exactly how each test case performed and model's reasoning
• Each iteration should show measurable improvement in evaluation scores
• Systematic approach: make one change at a time, evaluate impact, build on what works
• Final validation can use higher number of test cases after proven approach

Being Clear and Direct
• First line of prompt is the most important part of the entire request
• Clarity: Use simple language that anyone can understand
• Clarity: State exactly what you want without beating around the bush
• Clarity: Lead with straightforward statement of Claude's task
• Directness: Use instructions, not questions
• Directness: Start with direct action verbs like "Write," "Create," or "Generate"
• Bad example: "I need to know about those things people put on their roofs that use sun"
• Good example: "Write three paragraphs about how solar panels work"
• Bad example: "I was reading about renewable energy. What countries use geothermal?"
• Good example: "Identify three countries that use geothermal energy. Include generation stats for each."
• Application: Instead of "What should this person eat?" use "Generate a one-day meal plan for an athlete that meets their dietary restrictions"
• Immediate improvement possible: Clear opening line can significantly jump evaluation scores (e.g., 2.32 to 3.92)
• Key: Treat Claude like a capable assistant who needs clear direction, not someone who has to guess

Being Specific
• One of most effective ways to improve results is to be specific about what you want
• Provide clear guidelines or steps that direct Claude toward desired output
• Example: "Write a short story about a character who discovers a hidden talent" is too vague
• Vague prompts lead to countless possible directions and inconsistent results
• Adding specific guidelines gives Claude a clearer target and improves consistency and quality
• Output Quality Guidelines: Focus on qualities the output should have
• Quality guidelines control: Length, structure, format, specific attributes/elements, tone/style
• Quality guidelines example: Under 1,000 words, clear action revealing talent, supporting character
• Process Steps: Provide specific steps for Claude to follow systematically
• Process steps useful for: Complex problems, troubleshooting, decision-making, critical thinking
• Process steps example: 
1) Brainstorm talents, 
2) Pick most interesting, 
3) Outline pivotal scene, 
4) Brainstorm supporting characters
• Real-world impact: Specificity more than doubled scoresmore or less  (3.92 to 7.86)
• Always use output guidelines in almost every prompt as safety net for consistent results
• Add process steps for complex/ambiguous scenarios to ensure comprehensive consideration
• Professional prompts often combine both quality guidelines and process steps for best results

• Create clear boundaries and delimiters to help Claude understand prompt structure

Providing Examples
• One of the most effective prompt engineering techniques
• Known as "one-shot" (single example) or "multi-shot" (multiple examples) prompting
• Give Claude sample input/output pairs to guide responses
• Particularly useful for capturing corner cases or edge scenarios
• Useful for: Defining complex output formats, showing desired style/tone, handling ambiguous inputs
• One-Shot: Provide single example to establish pattern
• Multi-Shot: Provide multiple examples for different scenarios and edge cases
• Sarcasm example: A sarcastic tweet may appear positive but is actually negative
• Example with sarcasm handling: Include positive example, sarcastic example with explanation
• Always use XML tags to structure examples clearly (e.g., , )

• Use highest-scoring input/output pairs as examples from your evaluations
• Helps Claude understand what "perfect" output looks like for specific use case
• Add context to examples: Explain why output is good, not just format
• Example context: "This example is well-structured, provides detailed info, aligns with goals/restrictions"
• Best practices: Use XML tags, be explicit about what you're showing, address failure cases, explain ideal outputs

Tool use with Claude

 
1. Introducing tool use

Tools allow Claude to access information from the outside world, extending its capabilities beyond training data. Tool use follows a specific back-and-forth pattern:

- Initial Request: You send Claude a question with instructions on how to get extra data from external sources
- Tool Request: Claude analyzes the question and decides it needs additional information
- Data Retrieval: Your server runs code to fetch the requested information from external APIs or databases
- Final Response: You send the retrieved data back to Claude, which then generates a complete response

Key Benefits:
- Real-time Information: Access current data not available during training
- External System Integration: Connect Claude to databases, APIs, and other services
- Dynamic Responses: Provide answers based on latest available information
- Structured Interaction: Claude knows exactly what information it needs

 
 2. Tool Functions

A tool function is a plain Python function that gets executed automatically when Claude decides it needs extra information.

Best Practices:
- Use descriptive names: Both function and parameter names should clearly indicate purpose
- Validate inputs: Check that required parameters aren't empty or invalid
- Provide meaningful error messages: Claude learns from errors and might retry with corrected parameters

Example: get_current_datetime function accepts a date format parameter so Claude can request time in different formats.

 3. Tool Schemas

After writing a tool function, create a JSON schema that tells Claude what arguments your function expects. JSON Schema acts as documentation that Claude reads to understand when and how to call your tools.

Complete tool specification has three main parts:
- name: Clear, descriptive name for your tool (e.g., "get_weather")
- description: What the tool does, when to use it, and what it returns
- input_schema: Actual JSON schema describing function arguments

Writing Effective Descriptions:
- Aim for 3-4 sentences explaining what the tool does
- Describe when Claude should use it
- Explain what kind of data it returns
- Provide detailed descriptions for each argument

Easy Way to Generate Schemas: Use Claude itself to generate them by providing your tool function code and Anthropic documentation on tool use.

 4. Handling Message Blocks

When working with Claude's tool functionality, you encounter a new type of response structure - multi-block messages that contain both text and tool usage information instead of simple text responses.

Making Tool-Enabled API Calls:
Include a "tools" parameter in your API call with a list of JSON schemas describing available functions Claude can call.

Understanding Multi-Block Messages:
When Claude decides to use a tool, it returns an assistant message with multiple blocks:
- Text Block: Human-readable text explaining what Claude is doing
- ToolUse Block: Instructions for your code about which tool to call and what parameters to use
  	- An ID for tracking the tool call
  	- The name of the function to call
  	- Input parameters formatted as a dictionary
  	- The type designation "tool_use"

Managing Conversation History:
Claude doesn't store conversation history - you manage it manually. When working with tool responses, preserve the entire content structure including all blocks.

The Complete Tool Usage Flow:
1. Send user message with tool schema to Claude
2. Receive assistant message with text block and tool use block
3. Extract tool information and execute the actual function
4. Send tool result back to Claude along with complete conversation history
5. Receive final response from Claude

Updating Helper Functions:
If using helper functions like add_user_message() and add_assistant_message(), update them to handle multi-block content structures including tool use blocks.


 5. Sending Tool Results

After Claude requests a tool call, you need to execute the function and send the results back. This completes the tool use workflow.

Running the Tool Function:
When Claude responds with a tool use block, extract input parameters using response.content[1].input. This gives a dictionary of arguments. Use Python's unpacking syntax to call the function: get_current_datetime(response.content[1].input)

Tool Result Block:
After running the tool, send results back using a tool result block inside a user message:
- tool_use_id: Must match the ID of the ToolUse block
- content: Output from running your tool (as string)
- is_error: True if an error occurred

Handling Multiple Tool Calls:
Claude can request multiple tool calls in a single response. Each tool call gets a unique ID, and you must match these IDs when sending results back.

Building the Follow-up Request:
Your follow-up request must include:
- Original user message
- Assistant message with tool use block
- User message with tool result block

Making the Final Request:
When sending the follow-up, still include the tool schema even though Claude won't make another tool call. Claude needs it to understand tool references in conversation history. Claude then responds with a final message incorporating tool results into a natural response.


 6. Multi-turn Conversations with Tools

Multi-turn conversations with tools extend Claude's capabilities across multiple exchanges. Claude can make tool calls in one turn, receive results, and use that information in subsequent responses. This enables complex workflows where tool results inform further analysis and decision-making. Key aspects include managing conversation state, tracking which tools have been called, and providing updated context for Claude to make informed decisions.

 7. Implementing Multiple Turns

Implementing multiple turns requires building a loop that continues until the task is complete. The pattern involves: send messages to Claude, check if tool calls are requested, execute tools if needed, send results back, and repeat until Claude provides a final response without tool requests. This creates an agentic loop where Claude can repeatedly request information and refine its responses based on results.

 8. Using Multiple Tools

Using multiple tools allows Claude to choose from several available tools and compose them effectively. Claude can call different tools based on the user request, sequence tool calls logically, and even parallelize calls when appropriate. Provide multiple tool schemas in the tools parameter, each describing a different function. Claude learns when to use each tool based on descriptions and context.

 9. The Batch Tool

The batch tool enables running many tool invocations in bulk to improve throughput for high-volume tasks. Instead of making individual API calls for each tool invocation, batch operations allow you to group multiple tool calls together, reducing latency and improving efficiency. This is particularly useful when you have many independent tool calls that can be processed in parallel.

 10. Tools for Structured Data

Tools for structured data integrate tools that read, transform, or generate structured data like JSON. This ensures more reliable downstream processing by working with well-defined data structures. Tools can validate input, transform between formats, and ensure data consistency throughout the tool use workflow.

 11. Fine Grained Tool Calling

Fine grained tool calling enables controlling which tools are available when, constraining arguments, and tightening safety and guardrails for sensitive operations. You can dynamically enable or disable specific tools based on context, restrict argument values, and add validation to ensure proper tool usage. This provides granular control over Claude's tool interactions.

 12. The Text Edit Tool

The text edit tool is specialized for applying precise edits to text, such as code refactors or document revisions. Claude can provide specific edit instructions that the tool applies, enabling careful modifications to existing text without rewriting entire documents. This is useful for collaborative editing, code refactoring, and targeted content updates.

 13. The Web Search Tool

The web search tool allows Claude to query the web, retrieve external information, and ground answers in up-to-date search results. Claude can formulate search queries, analyze results, and integrate current information into responses. This is essential for handling questions about recent events, current information, and topics that change frequently.


Retrieval Augmented  
1. Introducing Retrieval Augmented Generation

RAG is a technique for working with large documents that are too big to fit into a single prompt. It breaks documents into chunks and only includes the most relevant pieces when answering questions.

Problems with Large Documents:
- Hard limit on prompt length
- Claude becomes less effective with very long prompts
- Larger prompts cost more to process
- Larger prompts take longer to process

RAG Solution:
- Break documents into smaller chunks during preprocessing
- When a user asks a question, find and include only the most relevant chunks
- Claude can focus on relevant content	
- Scales to very large documents and multiple documents
- Smaller prompts cost less and run faster

Challenges:
- Requires preprocessing to chunk documents
- Need a search mechanism to find relevant chunks
- Included chunks might not contain all context Claude needs
- Many ways to chunk text with different trade-offs

 2. Text Chunking Strategies

Text chunking strategies determine how to break large documents into manageable pieces. Different approaches include:

- Fixed-size chunks: Split documents into equal-sized pieces (e.g., 512 tokens)
- Structural chunking: Break at document boundaries (headers, paragraphs, sections)
- Semantic chunking: Split based on meaning and context
- Overlapping chunks: Include overlap between chunks to maintain context

Each strategy has trade-offs between context preservation, chunk size, and retrieval efficiency. The best approach depends on document structure and query patterns.


 3. Text Embeddings

Text embeddings convert chunks into numerical representations (vectors) that capture semantic meaning. Embeddings allow finding similar chunks through vector similarity rather than keyword matching. You can use pre-trained embeddings from APIs or libraries, store them in a vector database, and compare new queries to chunk embeddings to find the most relevant matches.

 4. The Full RAG Flow

The full RAG flow consists of: 
1) Preprocess documents into chunks, 
2) Generate embeddings for each chunk, 
3) Store chunks and embeddings in a database, 
4) When a query arrives, generate query embedding, 
5) Find most similar chunks using vector similarity, 
6) Include retrieved chunks in prompt to Claude, 
7) Claude uses the chunks to answer the question. This end-to-end process enables efficient retrieval of relevant information at scale.

 5. Implementing the RAG Flow

Implementing RAG involves writing code to: 
1) Load and parse documents, 
2) Split into chunks, 
3) Generate embeddings for chunks, 
4) Store in vector database (e.g., Pinecone, Weaviate), 
5) Create a search function that finds relevant chunks, 
6) Construct prompts with retrieved chunks, 
7) Pass to Claude for answer generation. Libraries like LangChain simplify this workflow.

 6. BM25 Lexical Search

BM25 is a lexical (keyword-based) search algorithm that ranks documents based on keyword matching. Unlike embeddings which use semantic meaning, BM25 matches exact keywords weighted by term frequency and document frequency. It's fast, interpretable, and works well for specific keyword queries. Can be used alongside semantic search for hybrid retrieval approaches. Basically it is an improved version of TF-IDF and has 2 extra params K,b

 7. A Multi-Index RAG Pipeline

A multi-index RAG pipeline combines multiple retrieval methods (semantic search, BM25 lexical search, metadata filtering) (RRF- reciprocal rank Fusion )to find the most relevant chunks. Uses different indexes optimized for different query types. Combines results from multiple methods to get better coverage. Useful for complex queries requiring both keyword and semantic matching.

 8. Reranking Results

Reranking results applies a more sophisticated model to reorder retrieval results after initial retrieval. An initial retriever finds many candidates, then a reranker scores them more accurately using cross-encoder models or LLMs. Improves quality of final retrieved chunks by filtering out less relevant ones while keeping the most important context.

 9. Contextual Retrieval

Contextual retrieval adds surrounding context to chunks for better understanding. Instead of retrieving just a small chunk, also include context from before and after to help Claude understand the full picture. Addresses the issue where isolated chunks might lack necessary context for Claude to answer questions accurately



FEATURES OF CLAUDE - COMPREHENSIVE README

1. Extended Thinking

Extended thinking is Claude's advanced reasoning feature that provides transparency into the model's thought process. When enabled, Claude's response includes two parts: the thinking process (scratch paper) and the final answer.

Key Points:
- Gives model time to work through complex problems
- Shows reasoning process for transparency
- Better accuracy on difficult problems
- Trade-offs: Higher costs, increased latency, complex response handling

Implementation:
- Minimum thinking budget: 1024 tokens
- max_tokens must be greater than thinking_budget
- Not compatible with message pre-filling and temperature
- Response includes signature for security (prevents tampering)
- Sometimes returns redacted thinking blocks (flagged by safety systems)

When to Use:
- After optimizing prompts first
- For complex reasoning tasks that need extra capability
- When standard prompting doesn't meet accuracy requirements



2. Image Support

Claude's vision capabilities allow you to include images in messages for analysis. You can ask Claude to describe, compare, count objects, or perform complex visual analysis tasks.

Limitations:
- Up to 100 images per request across all messages
- Max size: 5MB per image
- Single image: max 8000px height/width
- Multiple images: max 2000px height/width
- Images as base64 encoding or URL
- Token calculation: (width × height) / 750

Image Block Structure:
{
  "type": "image",
  "source": {
    "type": "base64",
    "media_type": "image/png",
    "data": image_bytes
  }
}

Prompting Techniques:
- Apply same prompt engineering techniques as text
- Provide detailed guidelines and analysis steps
- Use one-shot or multi-shot examples
- Break down complex tasks into smaller steps
- Simple prompts often lead to poor results

Real-World Example:
- Fire risk assessment for home insurance
- Analyzing satellite imagery for tree overhang
- Identifying emergency access routes
- Systematic analysis with structured prompts

3. PDF Support

Claude can read and analyze PDF files directly using nearly identical code to image processing, with a few key differences.

Key Changes from Image Processing:
- Type: "document" instead of "image"
- Media type: "application/pdf" instead of "image/png"
- Variable naming: file_bytes for clarity

Document Block Structure:
{
  "type": "document",
  "source": {
    "type": "base64",

  }

What Claude Can Extract:
- Text content throughout the document

- Document structure and formatting

Claude becomes a one-stop solution for extracting any type of information from PDFs - summaries, data analysis, or specific content extraction.

4. Citations

Citations allow Claude to reference specific parts of source documents, showing exactly where each piece of information comes from. This creates transparency and builds user trust.

Enabling Citations:
Add two fields to document block:
- title: Readable name for the document
- citations: {"enabled": True}

Citation Structure:
Each citation contains:
- cited_text: Exact text from document
- document_index: Which document (for multiple docs)

Citations with Plain Text:
Works with plain text sources too:
- Use "text/plain" media type
- Instead of page numbers, get character positions
- Pinpoints exact location in text

When to Use Citations:
- Users need to verify information
- Working with authoritative documents
- Transparency about sources is critical
- Users might want broader context around facts

Building UI with Citations:
- Create interactive elements (hover tooltips)
- Show source material directly
- Enable verification by checking original
- Transform Claude from "black box" to transparent assistant

5. Prompt Caching

Prompt caching speeds up responses and reduces costs by reusing computational work from previous requests. Instead of discarding preprocessing work, Claude saves and reuses it.

How It Works:
- Normal processing: tokenize, create embeddings, add context, generate output
- Without caching: all work discarded after response
- With caching: preprocessing stored for reuse
- Cache duration: 1 hour

Key Benefits:
- Faster responses (cached content executes quickly)
- Lower costs (pay less for cached portions)
- Automatic optimization (initial write, follow-ups read)

Limitations:
- Cache only lives 1 hour
- Only beneficial for repeated content
- Best for extremely frequent use of same content

Best Use Cases:
- Document analysis workflows
- Multiple questions about same large document
- Iterative editing with constant base content
- Consistent tool schemas across requests
- Stable system prompts

6. Rules of Prompt Caching

Cache Breakpoints:
- Caching NOT enabled automatically
- Must manually add cache breakpoint to blocks
- Work done BEFORE breakpoint gets cached
- Cache only used if content up to breakpoint is identical
- Use longhand form for text blocks with cache_control field

Cache Breakpoint Behavior:
- Caches all processing work up to and including breakpoint
- Content after breakpoint processed normally
- Even small changes (like adding "please") invalidate cache
- Entire cache needs reprocessing if content changes

Cross-Message Caching:
- Breakpoints can span multiple messages
- Breakpoint in later message includes all previous messages
- Useful for caching entire conversation context

What Can Be Cached:
- System prompts
- Tool definitions
- Image blocks
- Text blocks
- Tool use and tool result blocks

Cache Ordering:
Processing order: Tools → System prompt → Messages
Up to 4 cache breakpoints total

Minimum Content Length:
- Must be at least 1024 tokens to be cached
- Sum of all messages/blocks being cached
- Simple short messages won't meet threshold

Key to Effective Caching:
- Identify consistent content across requests
- Place breakpoints strategically
- Maximize reuse while minimizing invalidation


7. Prompt Caching in Action

Setup for Tool Schema Caching:
Create copies to prevent issues when adding cache control to last tool

System Prompt Caching:
Convert system prompt to structured format with cache_control field

Cache Behavior Patterns:
- First request: cache_creation_input_tokens (writes to cache)
- Follow-up: cache_read_input_tokens (reads from cache)
- Changed content: New cache creation tokens appear
- Single character change: Invalidates entire cache

Cache Ordering:
1. Tools (if provided)
2. System prompt (if provided)
3. Messages

Granular Caching:
- Change system prompt but keep tools = partial cache read (tools) + cache write (system)
- Only pay for processing parts that changed
- Cache is extremely sensitive to content changes

Practical Considerations:
- Most effective with consistent tool schemas
- Works best with stable system prompts
- Designed for frequent API usage (1 hour cache)
- Applications with multiple similar-context requests

8. Code Execution and the Files API

The Files API and Code Execution work together to enable powerful workflows where Claude can execute Python code on your data.

Files API:
- Alternative to encoding files as base64 in messages
- Upload files ahead of time, receive file ID
- Reference file ID in future messages
- Useful for files referenced multiple times or larger files

Code Execution Tool:
- Server-based tool (no implementation needed)
- Include predefined tool schema in request
- Executes Python in isolated Docker container
- No network access (can't make external API calls)
- Claude can execute code multiple times per conversation

Combining Both Features:
Typical workflow:
1. Upload data file (CSV, etc.) using Files API
2. Include container_upload block with file ID in message
3. Ask Claude to analyze the data
4. Claude writes and executes code to process file
5. Claude generates outputs (plots, reports) for download

Code Execution Environment:
- Isolated Docker container
- No network access
- Files API is primary way to get data in/out
- Multiple execution cycles per response

Response Structure:
- Text blocks: Claude's analysis
- Server tool use blocks: Actual code Claude runs
- Code execution tool result blocks: Output from code
- code_execution_output blocks: Generated file IDs for downloads

Downloading Generated Files:
Claude can create visualizations/reports and make them available via Files API

Beyond Data Analysis:
- Image processing and manipulation
- Document parsing and transformation
- Mathematical computations and modeling
- Report generation with custom formatting

Key Advantage:
Delegate complex computational tasks to Claude while maintaining control over inputs/outputs through Files API. Claude becomes a coding assistant that can actually execute and iterate on solutions.


Model Context Protocol (MCP)

MCP is a standard protocol for exposing tools, data sources (resources), and reusable prompts to LLMs through a server-client architecture. Key benefit: same MCP server can be reused across multiple apps and model providers without duplicating integrations.

Core Architecture:
- MCP Server: Process that implements MCP spec and exposes tools, resources, and prompts. Encapsulates business logic, external API calls, and database access.
- MCP Client: App hosting a model (or calling model API). Discovers servers, lists capabilities, surfaces tools/resources/prompts to the model, translates between model tool_calls and MCP invocations.
- Transport: Can use stdio (same machine), HTTP, WebSockets, or other protocols.

Message Types:
- ListToolsRequest/ListToolsResult: Client asks server "what tools do you have?"
- CallToolRequest/CallToolResult: Client requests server to run a specific tool
- ListResourcesRequest/ReadResourceRequest: Client fetches data/information
- ListPromptsRequest/GetPromptRequest: Client retrieves prompt templates

Key Building Blocks:

1. Tools:
- Typed operations model can call to "do things" (create tickets, run queries, send emails, etc.)
- Defined with name, description, input_schema (JSON Schema for parameters)
- Use Python SDK: @mcp.tool() decorator + type hints auto-generates schemas
- Example: read_document(doc_id), edit_document(doc_id, old_str, new_str)
- Best practice: Keep tools small, composable; let model chain multiple calls

2. Resources:
- Read-only or queryable data exposed via URIs (files, documents, DB rows, API endpoints)
- Two types: Direct resources (static URIs like docs://documents) and Templated resources (with parameters like docs://documents/{doc_id})
- SDK parses templated parameters and passes as keyword arguments
- Return with MIME type hints (application/json, text/plain, etc.)
- Use case: Document mentions (@doc_id), file browsing, lookups without tool calls

3. Prompts:
- Named, versioned prompt templates hosted on server + parameterized
- Return list of user/assistant messages that clients can feed directly to Claude
- Central place for prompt engineering; teams can update without redeploying clients
- Example: format_document(doc_id), summarize_report(doc_id)

Project Structure:
- MCP Server (mcp_server.py): Implements tools, resources, prompts using Python SDK
- MCP Client (mcp_client.py): Connects to server, provides list_tools(), call_tool(), list_resources(), read_resource(), list_prompts(), get_prompt()
- Application (main.py): Uses client to get tools/prompts, sends to Claude, processes results

Development Workflow:
1. Define tools/resources/prompts on server using decorators
2. Start MCP inspector: mcp dev mcp_server.py
3. Connect in browser inspector to test tools, resources, prompts manually
4. Implement client methods to call server
5. Wire client into application to use with Claude

Best Practices:
- Keep business logic in server, UI/orchestration in client
- Write clear tool/resource/prompt descriptions for model understanding
- Test via inspector before involving Claude
- Use MCP when multiple apps need same integrations
- Resources for data, tools for actions, prompts for workflows

Common Patterns:
- GitHub/GitLab server: tools for repo operations, resources for code browsing
- Document manager: tools for CRUD, resources for search/listing, prompts for reformatting
- Data analyst: tools for query execution, resources for schema/metadata, prompts for analysis workflows

HOW COMPUTER USE WORKS

Tool Use Refresher:

2. Claude decides it needs to use tool to answer question
3. Claude responds with tool use request containing tool name and input parameters
4. Your server runs actual function and returns result
5. Send result back to Claude in tool result message

Example: Weather Query
- Question: "What's the weather in San Francisco?"

- Claude calls tool with location parameter
- Your code fetches weather data
- Return "Sunny" back to Claude

Computer Use: Same Flow, Different Tool
Key Insight:
- Computer use follows exact same pattern as regular tool use
- Implemented as a tool - very special one that can interact with desktop environment

Process:
1. Include tool schema providing computer interaction capabilities
2. Claude decides to use computer tool
3. Instead of running simple function, execute requested action in computing environment
4. Send result (like screenshot) back to Claude

AGENTS AND WORKFLOWS

 1. Agents and Workflows Overview

Workflows and agents are strategies for handling user tasks that can't be completed by Claude in a single request.

When to Use Workflows vs Agents:
- Use workflows when you can picture the exact flow or steps that Claude should go through to solve a problem, or when your app's UX constrains users to a set of tasks
- Use agents when you're not sure exactly what task or task parameters you'll give to Claude

Key Definitions:
- Workflows: A series of calls to Claude meant to solve a specific problem through a predetermined series of steps
- Agents: Give Claude a goal and a set of tools, expecting Claude to figure out how to complete the goal through the provided tools

Example: Image to CAD Workflow
Building a web app where users drag and drop an image of a metal part to create a STEP file:
1. Feed an image into Claude, asking it to describe the object
2. Based on the description, ask Claude to use the CadQuery library to model the object
3. Create a rendering
4. Ask Claude to grade the rendering against the original image. If there are issues, fix them

The Evaluator-Optimizer Pattern:
- Producer: Takes input and creates output (Claude using CadQuery to model the part)
- Grader: Evaluates the output against some criteria
- Feedback loop: If the grader doesn't accept the output, feedback goes back to the producer
- Iteration: The cycle repeats until the grader accepts the output


 2. Parallelization Workflows

The Problem with Complex Single Prompts:
Asking Claude to do too much in a single request without specific criteria leads to unreliable results.

A Better Approach: Parallelization
Split the task into multiple parallel requests, each focusing on one aspect:
- Send the same image to Claude multiple times simultaneously
- Each request includes specialized criteria for one material (metal, polymer, ceramic, etc.)
- Claude evaluates the part's suitability for each material independently
- Collect all analysis results and feed them into a final aggregation step

How Parallelization Workflows Work:
- Split a single task into multiple sub-tasks
- Run the sub-tasks in parallel
- Aggregate the results together
- The parallelized sub-tasks don't need to be identical - each can have specialized prompts, tools, or criteria

Benefits:
- Focused attention: Claude concentrates on one specific aspect at a time
- Easier optimization: You can improve and test prompts independently
- Better scalability: Adding new evaluations is straightforward
- Improved reliability: Reduces cognitive load on the AI model

When to Use Parallelization:
When you have a complex decision that can be broken down into independent evaluations.


 3. Chaining Workflows

What is Workflow Chaining?
Breaking down a large, complex task into smaller, sequential subtasks instead of asking Claude to do everything at once.

Example: Social Media Marketing Tool
Instead of one massive prompt, break it down into steps:
1. Find related trending topics on Twitter
2. Select the most interesting topic (using Claude)
3. Research the topic (using Claude)
4. Write a script for a short format video (using Claude)
5. Use an AI avatar and text-to-speech to create a video
6. Post the video to social media

Why Chain Instead of One Big Prompt?
- Focus: Claude can concentrate on doing one task well rather than juggling multiple requirements
- Split large tasks into smaller, non-parallelizable subtasks
- Optionally do non-LLM processing between each task
- Keep Claude focused on one aspect of the overall task

The Long Prompt Problem:
Even with detailed constraints in one massive prompt, Claude might still violate some rules (use emojis, mention AI authorship, sound unprofessional).

The Chaining Solution:
Step 1: Send initial prompt and accept that first result might not be perfect
Step 2: Make a follow-up request focusing specifically on fixing issues:
- Identify and remove locations where text identifies author as AI
- Find and remove all emojis
- Locate any cringey writing and replace with professional text

When to Use Chaining:
- Complex tasks with multiple requirements
- Claude consistently ignores some constraints in long prompts
- Need to process or validate outputs between steps
- Want to keep each interaction focused and manageable


 4. Routing Workflows

Purpose:
Solve the problem where different types of user requests need different handling approaches. Route incoming requests to specialized processing pipelines instead of using a one-size-fits-all prompt.

The Problem with Generic Prompts:
A single generic prompt can't handle both educational content ("programming") and entertainment content ("surfing") effectively.

Setting Up Content Categories:
Define different types of content your application might need to generate:
- Entertainment: High-energy, culturally relevant content with trendy language
- Educational: Clear, engaging explanations with relatable examples
- Comedy: Sharp, unexpected content with clever observations
- Personal vlog: Authentic, intimate content with conversational storytelling
- Reviews: Decisive, experience-based content highlighting strengths and weaknesses
- Storytelling: Immersive content using vivid details and emotional connection

How Routing Works in Practice:
Two-step process:
1. Categorization: Send the user's topic to Claude to categorize it into one of predefined genres
2. Specialized Processing: Use the category result to select appropriate prompt template and generate content

Routing Workflow Architecture:
- User input goes to a router component first
- The router categorizes the request using an initial Claude call
- Based on the category, input gets forwarded to one specific processing pipeline
- Each pipeline can have its own workflow, prompts, or tools optimized for that category
- User input only goes to one specialized pipeline, not all of them

When to Use Routing:
- Application handles diverse types of requests needing different approaches
- Can clearly define categories that cover your use cases
- Categorization step can be handled reliably by Claude
- Performance benefit of specialized processing outweighs routing overhead


 5. Agents and Tools

Key Concept:
Agents represent a shift from structured workflows. While workflows are perfect when you know exact steps, agents shine when you're not sure what those steps should be.

How Tools Make the Agent:
The real power of agents lies in their ability to combine simple tools in unexpected ways.

Example: Basic Datetime Tools
- get_current_datetime: Gets the current date and time
- add_duration_to_datetime: Adds time to a given date
- set_reminder: Creates a reminder for a specific time

Claude can chain these together for complex requests:
- "What's the time?" → calls get_current_datetime
- "What day of the week is it in 11 days?" → chains get_current_datetime + add_duration_to_datetime
- Setting a gym reminder next Wednesday → uses all three tools in sequence
- "When does my 90-day warranty expire?" → Claude asks when you purchased before calculating

Tools Should Be Abstract:
Provide reasonably abstract tools rather than hyper-specialized ones.

Claude Code Example - Generic Tools:
- bash: Run any command
- read: Read any file
- write: Create any file
- edit: Modify files
- glob: Find files
- grep: Search file contents

No specialized tools like "refactor code" or "install dependencies" - Claude figures out how to use basic tools to accomplish complex tasks.

Best Practice: Combinable Tools
Provide tools that Claude can combine in creative ways.

Social Media Video Agent Example:
- bash: Access to FFMPEG for video processing
- generate_image: Create images from prompts
- text_to_speech: Convert text to audio
- post_media: Upload content to social platforms

This enables both simple workflows and interactive experiences where the agent adapts based on user feedback.


 6. Environment Inspection

Core Principle:
Claude operates blindly - it needs to be able to observe and understand the results of its actions to work effectively.

Why Environment Inspection Matters:
Every time Claude performs an action (like clicking a button), it needs to receive feedback (like a screenshot) to understand what happened. From Claude's perspective, clicking could:
- Navigate to a new page
- Open a menu
- Trigger any number of changes

Without seeing the results, Claude has no way to understand success or the new state of the environment.

Reading Before Writing:
Before Claude can modify any file, it needs to understand the current contents. Always follow this pattern when building agents.

Example: When asked to add a new route to a Python file, Claude first reads the existing code to understand the current structure before making changes.

System Prompts for Environment Inspection:
For complex tasks like video generation, include instructions like:
- Use bash tool to run whisper.cpp and generate caption files with timestamps to verify dialogue placement
- Use FFmpeg to extract screenshots from the video at regular intervals to visually inspect output
- Compare generated content against original requirements

Benefits of Environment Inspection:
- Better progress tracking: Claude can gauge how close it is to completing a task
- Error handling: Unexpected results can be detected and corrected
- Quality assurance: Output can be verified before considering task complete
- Adaptive behavior: Claude can adjust its approach based on what it observes

Practical Implementation:
Always ask: "How will Claude know if this action worked?"
- Reading file contents before modifications
- Taking screenshots after UI interactions
- Checking API responses for expected data
- Validating generated content against requirements


 7. Workflows vs Agents - Comparison

What Are Workflows?
A predefined series of calls to Claude designed to solve a known problem or set of problems. Use workflows when you can picture the flow of steps ahead of time.
- Break down big tasks into smaller, more specific subtasks
- Each step focuses on a single area for more precise work

What Are Agents?
Claude gets a set of basic tools and is expected to formulate a plan to use these tools to complete a task.
- Don't know exactly what tasks will be provided
- System needs to be more adaptive
- Can creatively figure out how to handle challenges by combining tools in unexpected ways

BENEFITS OF WORKFLOWS:
- Claude can focus on one subtask at a time, leading to higher accuracy
- Far easier to evaluate and test (know each exact step)
- More predictable and reliable execution
- Better suited for solving specific, well-defined problems

BENEFITS OF AGENTS:
- Allow for more flexible user experience
- Far more flexible task completion - Claude can combine tools in unexpected ways
- Can handle novel situations not anticipated during development
- Can ask users for additional input when needed

DOWNSIDES OF WORKFLOWS:
- Far less flexible - dedicated to solving specific types of tasks
- Generally more constrained user experience - need to know exact inputs to the flow
- Require more upfront planning and design work

DOWNSIDES OF AGENTS:
- Lower successful task completion rate compared to workflows
- More challenging to instrument, test, and evaluate (don't know what series of steps agent will execute)
- Less predictable behavior

WHEN TO USE EACH APPROACH:
Your primary goal as an engineer is to solve problems reliably. Users don't care about fancy agents - they want products that work consistently.

General Recommendation:
Always focus on implementing workflows where possible, and only resort to agents when they are truly required.
- Workflows: Provide the reliability and predictability most production applications need
- Agents: Offer flexibility for scenarios where exact requirements can't be predetermined

Consider workflows when: You have well-defined processes
Consider agents when: You need to handle unpredictable, varied user requests requiring creative problem-solving
